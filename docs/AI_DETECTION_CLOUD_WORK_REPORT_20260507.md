## 一、工作主题

本阶段围绕“车端上传图片到云端，云端完成 AI 检测，再把结果返回车端或回写到网页端”这一条主线展开。

这条链路不是单一模型，而是分成了三个连续阶段：

1. 早期的 `GroundingDINO` 车端检测基础能力和 `engine` 生成方案。
2. 2026 年初逐步落地的 `Qwen3-VL-2B` 云端 Yes/No 复核链路。
3. 2026 年 5 月新增的 `Qwen3.6-27B-MM` 云端多模态大模型检测链路。

从职责上看，这一块既有车端侧工作，也有云端侧工作。我在里面承担的核心角色，不是简单“调个网页”，而是分阶段把整条检测链的关键环节做起来：

1. 早期负责 `GroundingDINO` 的 `engine` 生成方案，解决车端推理落地的关键前提。
2. 中期把部分车端页面和业务逻辑交给他人继续推进，但我保留并主导了云端 AI 检测扩展方向。
3. 后期主导云端 `Qwen3-VL-2B` 检测链和 `Qwen3.6-27B-MM` 多模态检测链的接入、网页化和归档化。

## 二、总体架构

当前 AI 检测相关能力，实际分成两条链：

### 1. 车端 GroundingDINO + 云端 Qwen3-VL-2B 复核链

主链路：

```text
车端相机 / 车端业务
  -> GroundingDINO 检测或候选框生成
  -> 车端通过 WebSocket 上传到云端
  -> 云端 qwen_ws_checker_service
  -> 本机 Qwen3-VL-2B vLLM
  -> 返回 Yes / No 检测结果
  -> 回传车端
  -> 同时归档图片、请求、结果、ROI
```

这一条链解决的是：

1. 车端先做轻量候选目标发现。
2. 云端再用多模态大模型做二次判断。
3. 形成“车端检测 + 云端复核”的端云协同方案。

### 2. 网页 / 云端 Qwen3.6-27B-MM 检测链

主链路：

```text
网页上传图片
  -> jgzj backend /api/qwen36-mm-check
  -> 本机 127.0.0.1:18001 tunnel
  -> A100 127.0.0.1:8001 Qwen3.6-27B-MM
  -> 返回 Yes / No
  -> 网页展示结果
  -> 同时写入 AI 检测归档
```

这一条链当前更偏：

1. 云端高阶模型验证
2. 网页演示和效果复核
3. 作为后续替换或升级 `Qwen3-VL-2B` 的预研路径

需要明确的是：

- `Qwen3-VL-2B` 这条链，已经是“车端上传 -> 云端判断 -> 结果回车端”的实际业务链路。
- `Qwen3.6-27B-MM` 这条链，目前已经打通云端检测和网页体验，但当前更偏云端验证与高质量复核入口，不等同于已经全面替换车端现有主链。

## 三、阶段一：早期 GroundingDINO engine 方案建设

### 1. 主要职责

在这一阶段，我最核心的贡献，不是 UI，而是 `GroundingDINO` 的 `engine` 生成方案。

这件事的重要性在于：

1. 车端如果没有稳定可用的 `engine`，就很难在实际设备上把检测链跑起来。
2. 只有把 `engine` 生成方案掌握住，后续车端部署、迁移、复制、性能优化才有抓手。
3. 这是整个车端 AI 检测能力真正“能落地”的基础。

### 2. 工作内容

我当时主要负责：

1. 梳理 `GroundingDINO` 车端推理落地所需的 `engine` 生成流程。
2. 解决 `engine` 文件生成的关键工程问题。
3. 形成可复现的 `DINO engine` 生成方案，而不是一次性手工试验。
4. 让这套方案可以支撑后续车端业务侧继续开发。

### 3. 阶段意义

这部分虽然不一定最容易在网页上看见，但它是后面所有检测链的前提条件。

如果没有这个阶段的工作，后面很多事情都很难推进：

1. 车端无法稳定做候选检测或框选。
2. 云端复核链也没有高质量输入。
3. 整个方案只能停留在实验室，无法走向真实业务链。

### 4. 汇报口径

这一部分在述职中应该明确表述为：

> 我比较早期负责了 GroundingDINO 在车端落地所需的 engine 生成方案，这部分非常关键。它不是简单脚本调用，而是把后续车端检测能力真正跑起来的基础工程。我掌握了 DINO engine 的生成方法，解决了车端部署前提问题。

## 五、阶段三：云端 Qwen3-VL-2B Yes/No 复核链落地

### 1. 背景

仅靠车端规则或轻量检测模型，很多场景会出现：

1. 误报
2. 漏报
3. 语义判断不稳定
4. 对复杂事件缺乏更强的视觉理解能力

所以需要一层云端复核，把车端上传的图像或 ROI，再交给多模态模型做二次判断。

### 2. 我的工作目标

这一阶段我做的，不是单纯跑个 `vLLM`，而是把云端复核能力真正产品化、服务化：

2. 让云端模型以统一格式返回 `YES / NO`。
3. 让结果可回车端。
4. 让所有请求、原图、ROI、响应结果都可归档。
5. 让这条能力能在网站里被查看、回放和演示。
